# 大陆商业化运营路线图

这份路线图按“先能访问，再能赚钱，再能上架”的顺序设计。它不是法律意见，正式上线前建议找云服务商备案专员、广告/支付渠道经理和游戏合规律师确认。

## 目标

- 中国大陆手机浏览器和 PC 浏览器稳定访问。
- 游戏作为流量入口，支持广告变现。
- 后续支持充值、账号、云存档和活动。
- 保留游戏编写进度和版本记录。
- 未来可迁移到微信小游戏、抖音小游戏、快应用、App 或其他平台。

## 阶段 1：大陆可访问

当前静态站可以作为原型，但正式站应迁移到国内基础设施：

1. 购买域名，主体建议与后续收款主体一致。
2. 选择国内云：阿里云、腾讯云、华为云、火山引擎等。
3. 完成 ICP 备案，页面底部展示备案号并链接到工信部备案系统。
4. 完成公安联网备案，页面底部展示公安备案号。
5. 使用国内 CDN 和 HTTPS 证书。
6. 移除所有海外依赖：海外字体、海外统计、海外广告、海外脚本、海外图片。

## 阶段 2：广告变现

广告不要先写死到游戏里，要做成可配置模块：

- 首页广告位：游戏列表上方、列表中间、底部。
- 游戏页广告位：开始前激励广告、结算页广告、底部横幅。
- 广告配置：独立 `ad-config`，支持开关、频控、平台 ID、测试模式。
- 当前占位文件：`data/ad-config.json`。
- 合规展示：广告必须显著标识，不能伪装成游戏按钮。
- 体验控制：不要在第一局开屏强弹，先让玩家玩到爽点再展示激励广告。

推荐优先评估国内广告平台或小游戏平台自带广告能力。不要依赖 Google AdSense 这类大陆访问不稳定的方案。

## 阶段 3：充值与账户

充值不是静态站能安全完成的，需要后端。建议后端最小模块：

- 用户系统：手机号/微信/平台账号登录。
- 实名认证与未成年人识别。
- 订单系统：创建订单、支付状态、发货状态、退款状态。
- 支付：微信支付、支付宝或平台内支付。
- 商品：会员、去广告、皮肤、复活币、通行证等。
- 云存档：分数、成就、道具、购买记录。
- 风控：刷单、重放回调、异常充值、退款滥用。
- 对账：每日支付渠道对账和运营后台。

未完成实名、防沉迷、版号和支付主体准备前，不建议上线充值。

## 阶段 4：游戏合规

如果只是内部测试或朋友试玩，风险较低。若面向公众运营，尤其是接广告、充值、账号体系或平台上架，要按网络游戏运营来评估：

- 游戏内容审查：政治、色情、暴力、赌博、迷信、诱导未成年人等内容风险。
- 版号/出版审批：公开运营网络游戏通常需要提前确认。
- 实名认证：不能向未实名用户提供网络游戏服务。
- 未成年人防沉迷：按监管要求限制服务时间。
- 未成年人充值：限制金额、退款和监护人处理机制。
- 用户协议、隐私政策、儿童个人信息规则。

## 阶段 5：多平台上架

保留一套核心游戏逻辑，按平台适配外壳：

- Web/H5：当前站点。
- 微信小游戏：适配小游戏 API、广告、支付、登录、存档。
- 抖音小游戏：适配平台登录、广告、支付、榜单。
- App：适配 Android/iOS 包体、商店审核、App 备案、SDK 合规。
- PC 桌面：Electron 或轻量壳，适合后续发行平台。

建议从现在开始把游戏拆成：

- `core`: 游戏规则、碰撞、计分。
- `renderer`: Canvas/WebGL 渲染。
- `platform`: 广告、支付、登录、存档、分享。
- `content`: 关卡、配置、文案、皮肤。

## 技术架构建议

短期：

- 静态前端：国内 CDN。
- 配置清单：`data/games.json`。
- 本地进度：`PROJECT_PROGRESS.md`。

商业化：

- API 后端：Node.js / Go / Java 均可。
- 数据库：MySQL/PostgreSQL。
- 缓存：Redis。
- 对象存储：游戏资源、封面、音频。
- 日志：访问、错误、支付、广告回调。
- 后台：游戏管理、广告配置、商品配置、订单查询、用户封禁。

## 不要做的事

- 不要把充值做成前端直接跳转收款码。
- 不要用海外免费 Pages 当大陆商业主站。
- 不要接来源不明的广告脚本。
- 不要在未备案域名上长期对大陆用户运营。
- 不要收集身份证、手机号、定位等信息后没有隐私政策和安全措施。
- 不要在没有合规确认前宣传“永久免费提现”“赚钱游戏”“抽奖返现”等高风险玩法。
